# 微信手机号授权登录正式设计文档

## 1. 文档目的

本文用于明确 `前呈似景智链` 小程序从“手填手机号登录”升级为“微信手机号授权登录”的正式改造方案，作为：

- 登录安全改造的实现依据
- 上线前安全收口的评审文档
- 后续开发、测试、提审时的统一口径

本文关注的是**正式登录方案**。联调期保留的临时能力仅作为过渡，不作为正式产品能力。

## 2. 背景

当前项目的登录方式为：

1. 用户进入登录页
2. 手动输入手机号
3. 前端调用登录接口
4. 后端按手机号查找或创建用户

当前相关代码位置：

- 登录页：
  - `pages/login/login.js`
  - `pages/login/login.wxml`
- 前端鉴权：
  - `utils/auth.js`
- 云函数接口：
  - `cloudfunctions/api/index.js`
  - `authLoginByPhone`
  - `authGetCurrentUser`

该方案便于开发联调，但不满足正式上线要求。

## 3. 问题陈述

### 3.1 安全问题

当前方案的核心问题是：

**“知道手机号”不等于“拥有手机号”。**

也就是说，任何人只要知道某个客户或管理员的手机号，就有机会尝试登录该身份。

### 3.2 具体风险

#### 普通客户侧风险

- 可能冒用他人身份查看价格
- 可能访问他人订单、地址、客户资料
- 可能影响客户数据可信性

#### 管理后台风险

- 若手机号命中管理员账号，存在冒充后台身份的风险
- 后台涉及商品价格、库存、审核、对账等敏感数据，风险级别更高

#### 合规与审核风险

- 无法向业务方证明“手机号属于当前操作者”
- 不利于通过正式安全评估
- 作为正式版本提交审核时，存在明显的安全短板

## 4. 改造目标

将登录方式升级为：

**用户通过微信小程序官方手机号授权能力完成登录，后端基于当前微信身份与授权手机号完成用户识别、建档和角色同步。**

目标包括：

1. 正式环境不允许手填手机号直接登录
2. 手机号必须来自微信授权链路
3. 用户档案与当前微信 `OPENID` 绑定
4. 管理员角色在“已验证手机号”的前提下同步
5. 保持当前业务流程不变：
   - 登录
   - 看价申请
   - 地址
   - 选购
   - 订单
   - 后台权限

## 5. 设计范围

### 本次改造范围

- 登录页交互
- 前端鉴权工具
- 云函数登录接口
- 用户表登录相关字段
- 管理员角色识别链路
- 正式/联调开关策略
- 测试与提审口径

### 本次不改范围

- 申请看价业务规则
- 订单、地址、审核、对账的数据结构
- 微信支付能力
- 管理员角色模型本身

## 6. 总体方案

### 6.1 正式登录链路

正式版本改为：

1. 用户进入登录页
2. 点击“微信手机号一键登录”
3. 小程序触发微信手机号授权能力
4. 前端拿到微信返回的授权凭据（`code`）
5. 前端将该凭据提交给云函数
6. 云函数基于当前微信身份和微信官方手机号能力换取手机号
7. 后端按手机号查找或创建用户
8. 后端将当前 `OPENID` 绑定到该用户
9. 若手机号命中 `admins` 表，同步管理员角色
10. 返回当前用户，完成登录

### 6.2 过渡期策略

为了不打断当前联调，保留一条**仅开发可用**的临时登录方式：

- 手填手机号登录
- 默认隐藏
- 提审版本关闭
- 正式版本关闭

## 7. 前端改造设计

## 7.1 登录页改造

### 当前状态

当前登录页使用：

- 手机号输入框
- “登录”按钮

### 改造后状态

正式方案改为：

- 主按钮：`微信手机号一键登录`
- 按钮能力：`open-type="getPhoneNumber"`
- 手填手机号区域默认不展示

### 建议页面结构

1. 标题：`前呈似景智链`
2. 说明文案：`授权微信手机号后可申请查看商品价格`
3. 主按钮：`微信手机号一键登录`
4. 次要说明：拒绝授权时的说明文案
5. 仅开发模式下可见的联调入口（隐藏在正式环境）

### 交互要求

- 授权成功：
  - 显示 loading
  - 自动完成登录
  - 跳回原页面或“我的”
- 用户拒绝授权：
  - toast：`需完成手机号授权后才可登录`
- 网络或后端异常：
  - toast：`登录失败，请稍后重试`

## 7.2 登录后跳转

沿用现有规则即可：

- 有 `redirect`：回原业务页
- 无 `redirect`：跳“我的”

这里不需要重新设计，只需要让新登录链继续接入现有跳转逻辑。

## 7.3 联调开关

建议在运行时配置中加入明确开关，例如：

- `wechatPhoneAuthEnabled`
- `devPhoneLoginEnabled`

建议规则：

- 开发联调：
  - `wechatPhoneAuthEnabled = true`
  - `devPhoneLoginEnabled = true`
- 提审版本：
  - `wechatPhoneAuthEnabled = true`
  - `devPhoneLoginEnabled = false`
- 正式版本：
  - `wechatPhoneAuthEnabled = true`
  - `devPhoneLoginEnabled = false`

## 8. 后端改造设计

## 8.1 云函数接口改造

### 当前接口

- `authLoginByPhone`
- `authGetCurrentUser`

### 目标接口

建议新增：

- `authLoginByWxPhone`

并将：

- `authLoginByPhone`

降级为**开发联调专用接口**。

## 8.2 新接口输入输出

### 输入

- `phoneCode`

说明：
- 由前端通过微信手机号授权组件拿到
- 正式登录不再直接传明文手机号

### 输出

- 当前用户档案
- 当前角色信息
- 是否命中管理员

## 8.3 后端处理流程

1. 通过 `cloud.getWXContext()` 获取当前 `OPENID`
2. 使用微信手机号授权凭据换取手机号
3. 按手机号查询 `users`
4. 按手机号查询 `admins`
5. 若 `users` 不存在：
   - 创建新用户
6. 若 `users` 已存在：
   - 更新 `openid`
   - 更新登录时间
7. 若手机号命中管理员：
   - 同步 `role`
8. 返回统一用户对象

## 8.4 旧接口处理策略

`authLoginByPhone` 不建议直接删除，建议分阶段处理：

### 阶段一

- 保留接口
- 仅开发模式可用

### 阶段二

- 正式环境禁用该接口
- 若外部误调，直接返回失败：
  - `仅开发联调环境可用`

## 9. 数据模型设计

### 9.1 当前可复用字段

`users` 已有字段：

- `id`
- `openid`
- `phone`
- `role`
- `updatedAt`

这些字段足以支撑第一阶段改造。

### 9.2 建议新增字段

建议在 `users` 中增加：

- `phoneAuthAt`
  - 最近一次微信手机号授权登录时间
- `phoneAuthSource`
  - 固定值建议：`wechat_phone`
- `lastLoginAt`
  - 最近登录时间

这些字段有助于后续：

- 排查登录问题
- 做安全审计
- 区分手填联调用户与正式授权用户

## 10. 权限与安全策略

### 10.1 正式环境禁止明文手机号直登

正式环境中，前端不应再把手机号直接作为登录凭据提交给后端。

### 10.2 后端只信微信授权手机号

正式登录时，手机号必须由微信官方能力返回，不能信任前端自填。

### 10.3 管理员也必须走同一认证链

管理员不应因为“手机号在管理员表里”就可以被手填冒用。  
必须满足：

- 当前微信身份存在
- 手机号经过微信授权验证
- 手机号命中管理员表

### 10.4 现有敏感接口继续按 `OPENID + role` 校验

当前云函数已经有这套方向，应继续沿用：

- 当前用户识别
- 管理员校验
- 超级管理员校验
- 订单 / 地址归属校验

## 11. 对现有业务的影响

### 11.1 用户侧业务

不会改变这些业务本身：

- 申请查看价格
- 地址管理
- 提交订单
- 订单列表与详情

变化只是：

- 登录更安全
- 身份绑定更可信

### 11.2 后台侧业务

不会改变这些业务结构：

- 商品管理
- 审核中心
- 对账中心

变化只是：

- 管理员入口不再依赖“手填管理员手机号”

## 12. 迁移方案

建议采用三阶段迁移。

### 阶段 1：并行接入

- 新增微信手机号授权登录接口
- 登录页增加微信授权主按钮
- 保留手填手机号联调入口

### 阶段 2：前端切主路径

- 登录页默认只展示微信手机号授权按钮
- 手填入口仅开发模式可见

### 阶段 3：正式收口

- 正式提审前关闭手填手机号入口
- 正式环境禁用 `authLoginByPhone`

## 13. 测试方案

### 13.1 用户侧

- [ ] 未登录进入受限页面会跳登录
- [ ] 点击微信手机号授权可登录成功
- [ ] 拒绝授权有明确提示
- [ ] 登录后能回到原页面
- [ ] 已通过客户登录后能看价格

### 13.2 管理侧

- [ ] 管理员手机号授权登录后可见后台入口
- [ ] 超级管理员手机号授权登录后可审核
- [ ] 普通客户不能因知道管理员手机号而冒充后台身份

### 13.3 安全侧

- [ ] 正式环境不显示手填手机号入口
- [ ] 正式环境 `authLoginByPhone` 不可用
- [ ] 所有敏感接口继续走 `OPENID + role` 校验

## 14. 风险与注意事项

### 14.1 微信授权依赖平台能力

该方案依赖微信小程序官方手机号授权能力。  
上线前需再次核对微信官方平台当前要求、能力说明与可用性。

### 14.2 老用户迁移

当前测试和联调环境中，已有一批通过“手填手机号”创建的用户。  
改造后需要允许这些用户在首次微信授权登录时：

- 命中已有手机号
- 完成 `openid` 绑定
- 不重新创建重复账号

### 14.3 管理员手机号唯一性

`admins.phone` 应保持唯一，否则角色同步可能产生歧义。

## 15. 实施顺序建议

1. 在 `utils/config.js` 中补登录能力开关
2. 改造登录页 UI
3. 在 `utils/auth.js` 中新增微信手机号授权登录方法
4. 在 `cloudfunctions/api/index.js` 中新增 `authLoginByWxPhone`
5. 将 `authLoginByPhone` 收口为开发联调专用
6. 跑一轮普通客户 / 管理员 / 超级管理员回归
7. 提审前关闭手填手机号入口

## 16. 结论

“手填手机号登录”可以继续用于短期开发联调，但它属于：

- **正式上线前必须替换**
- **优先级很高的安全改造项**

本项目的正式登录方案应统一切换为：

**微信手机号授权登录 + `OPENID` 绑定 + 管理员手机号映射校验**
